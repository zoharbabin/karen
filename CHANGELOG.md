# Changelog

All notable changes to the Karen plugin are recorded here. The eval benchmark (`evals/`) versions independently — see `EVALS-PLAN.md` §11.4.

Format based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/).

## [Unreleased]

### Added
- Repo scaffolding for external contribution and feedback: issue templates, PR template, `CONTRIBUTING.md`, this changelog, and a CI workflow running the eval benchmark's self-test suite.
- `evals/self-test/run-self-test.js` — an automated runner for the self-test contract (`CONTRACT.md` §4): every fixture's golden sample must pass every applicable grading dimension, and every broken sample must fail exactly its declared flaw(s).
- `.claude-plugin/marketplace.json` and `plugins/karen/.claude-plugin/plugin.json` — this repo now self-hosts as its own Claude Code plugin marketplace.
- `plugins/karen/skills/karen/SKILL.md` and 13 `reference/*.md` files — Karen ported from `BLUEPRINT.md` into a runtime-executable Claude Code skill: detection, the interview, gate generation and the gate contract, run-state/delta/circuit-breaker, every quality dimension, deployment-context profiles, monorepo scoping, reconciliation, integration, and voice.
- `plugins/karen/commands/init.md` and `commands/audit.md` — thin, discoverable slash-command triggers into the skill's procedures.
- `plugins/karen/commands/feedback.md` + `scripts/collect-feedback-bundle.sh` — opt-in, user-initiated feedback: gathers a redacted bundle (`.karen.json`, gate summary counts, plugin version, tool-probe results), shows it to the user, then files via `gh issue create` or a pre-filled URL. No automatic telemetry.

### Fixed
- `evals/schema/CONTRACT.md` §3 named a standalone `issue-id.js` helper that never existed; `issueId()` has always lived inside `parse-gate-output.js`. Doc corrected to match the code.
- `gate-contract.md`/`write-harness.md` never said how a non-shell gate (Node, Python) should actually be invoked, and `evals/runner/fixture-workflow.js`'s audit-sequence prompt hardcoded `bash <script>` regardless of the gate's real extension — a real `mode: 'full'` run surfaced this: `.mjs` gates ran fine under some sub-agents' own judgment but produced bash syntax-error garbage under others that followed the literal instruction, corrupting 2 of 7 audit triggers non-deterministically. Fixed by specifying direct-execution-via-shebang as the one invocation rule everywhere a gate gets run (`run_gate`, `run-all.sh`, the eval runner).
- `integration.md` referenced a `.karen/run-all.sh` orchestrator in six places as something `write_harness` already produces, but no reference file described it being generated. `write-harness.md` now documents it as a real output of `write_harness`.
- `probe-tools.md` checked a candidate tool's presence and config but never its actual flags — a dogfood run against a real project assumed `gitleaks --exclude-path` existed when the installed CLI doesn't support it. Added a step to verify any flag a gate depends on before shipping it.
- `evals/grading/score-gate-issues.js` matched emitted findings to planted issues by `file:line` alone, with no awareness of which categories a gate is even supposed to cover — an unrelated finding from one gate could land on the same line as a different gate's planted decoy and get misattributed as a false positive against a category it never touched. Matching is now scoped to each gate's `coversCategories` from `expected-gates.json`.
- `gate-contract.md` never called out that a wrapped tool that's missing, crashes, or errors out must not collapse into the same `PASS (0 issues)` as a tool that ran clean — a `|| true` written to survive the tool's own nonzero-exit-on-findings behavior also silently survives the tool never running at all. Added "Fail loudly when the wrapped tool itself didn't run" to `gate-contract.md` and ported it, along with several other confirmed `gate-contract.md`/`write-harness.md`/`probe-tools.md` rules that had drifted out of sync, into `BLUEPRINT.md`.
- `evals/fixtures/python-monorepo/patches/01-partial-fix.patch`'s replacement line exceeded ruff's 100-char limit, triggering an undeclared `gate-2-completeness` regression alongside the patch's intended `gate-3-security` fix. Split into two lines.
- `evals/fixtures/go-monorepo/patches/02-regression.patch`'s planted secret string's first 16 characters were low-entropy dictionary text, which gosec's G101 entropy-truncation heuristic scores below its detection threshold — reordered the string so the high-entropy hex prefix comes first.
- `evals/fixtures/go-mcp-server-single/expected-karen.json`'s `audience` field expected `"developer-tooling"`, a value that appears nowhere in `BLUEPRINT.md`'s documented audience examples (`enterprise`/`internal`) and contradicts the fixture's own `answer-key.md`, where the scripted user says "It's an **internal** developer tool, not customer-facing." Every other MCP-server fixture (Node, Python) correctly expects `"internal"`. Fixed the fixture's ground truth and its two self-test captures to match, restoring parity.
- `quality-dimensions.md`/`BLUEPRINT.md`'s known-gaps guidance told the agent to ask "known limitation, or work in progress?" but never said what a "work in progress" answer should do — a real `mode: 'full'` run showed an agent write a `knownGaps` entry for a stub whose own `reason` field admitted "still in progress, not a tracked limitation," silently suppressing a live gate-2-completeness finding the user still needed to see. Added the rule explicitly: "work in progress" gets no `knownGaps` entry at all.
- `write-harness.md`/`BLUEPRINT.md`'s existing "never nest a JSON-parsing one-liner in a nested quoting context" rule documented the exact failure mode almost verbatim, but a live re-run still reproduced it: the generated `gate-3-security.sh` embedded a Python one-liner with a backslash-escaped quote inside an f-string expression (`f"{r[\"filename\"]}..."`), which crashes with a `SyntaxError` on every invocation — confirmed by running the script directly and by verifying `bandit`/`python3` are both genuinely installed, ruling out an environment gap. Documenting a failure mode in prose didn't stop a live agent from writing it again, since nothing forced the agent to actually execute the gate before finishing. Added a new rule to both docs: run every gate script for real against the project root, and confirm it reports a genuine finding rather than a false `PASS (0 issues)`, before considering `write_harness` done.
- `BLUEPRINT.md`/`quality-dimensions.md`'s "Stub implementations" signal never said which line to report for a multi-line stub (declaration vs. the actual `throw`/`raise` statement) — a live-generated `gate-2-completeness.sh` for `python-mcp-server-single` matched on the `def page_on_call(...):` line and reported that line number instead of the `raise NotImplementedError(...)` line one below it, an off-by-one that would have looked like a fixture bug but traced directly to this doc gap once compared against every other stub-implementation fixture's ground truth (all of which expect the statement's own line). Added the rule explicitly: report the line of the offending statement itself, not the enclosing declaration.
- `evals/runner/fixture-workflow.js`'s audit-relay `StructuredOutput` schema only required each audit trigger's top-level key, not its nested `gateResults`/`runState` fields — a live relay call for `python-monorepo` silently omitted the `initial` trigger's `runState` entirely (still schema-valid with only `gateResults` present), which starved `score-delta.js` of the previous run's per-gate counts and produced a completely empty `partialFix.actualDeltas: {}` instead of a real delta or an explicit error. Fixed by marking `gateResults` and `runState` both `required` per-trigger in the schema.
- `evals/runner/fixture-workflow.js`'s `mode: 'full'` Karen-turn agent had unrestricted filesystem access to this whole repo, including each fixture's own grading ground truth — a re-run caught it reading `evals/self-test/golden/python-mcp-server-single/run-capture.json` mid-interview, declaring "this is exactly the same interview content as our conversation," and skipping straight to `done` without ever asking two documented "must ask unprompted" items (AI-powered classification, coverage threshold), instead copying the golden capture's values verbatim. Every prior `mode: 'full'` result, including fixtures already reported passing, ran under this same exposure and is unconfirmed until re-run. Fixed by explicitly forbidding the Karen-turn prompt from reading `evals/self-test/`, `answer-key.md`, `expected-karen.json`, `planted-issues.json`, `expected-gates.json`, or `fixture-manifest.json` for the fixture under test.

### Validated
- First real `mode: 'full'` end-to-end run against the installed Karen plugin (`node-sdk-single` fixture): 90.9% pass@1 across 11 grading dimensions, one failing dimension (`delta`) traced to the gate-invocation bug above. After the fix, a repeat run scores 100.0% pass@1 across all 11 dimensions. Real-project dogfooding (separately, against the same fixture) confirmed detection (4/4 planted issues caught, 0/3 decoys flagged), the interview's must-ask/must-not-ask behavior, and the gate contract's mechanics (tab-byte parsing, one-line-per-issue, zero-tolerance marking, tool-absence-fails-loud) all hold under real tool execution (npm, eslint, semgrep, gitleaks, vitest).
